# 앤서블 학습 로드맵

## 문서 목적
이 문서는 앤서블을 처음 배우는 사람부터 실무 자동화 아키텍처를 설계하는 전문가까지 단계적으로 성장할 수 있도록 만든 학습 계획서다. 각 단계는 학습 목표, 핵심 주제, 실습 과제, 산출물, 다음 단계 진입 기준으로 구성한다.

## 전체 학습 흐름
1. 초급: 앤서블의 기본 개념과 실행 구조 이해
2. 초중급: 플레이북과 변수, 조건문, 반복문을 이용한 자동화 구현
3. 중급: 역할 기반 구조화와 재사용 가능한 프로젝트 설계
4. 중고급: 운영 환경 분리, 보안, 템플릿, 에러 처리, 테스트 강화
5. 고급: 대규모 인프라 자동화와 배포 전략 설계
6. 전문가: 자동화 표준 수립, 플랫폼 연동, 조직 차원의 운영 체계 구축

## 0단계: 준비 단계
### 목표
- 앤서블 학습에 필요한 기본 운영체제, CLI, YAML, SSH 배경지식을 갖춘다.

### 핵심 주제
- 리눅스 기본 명령어
- SSH 접속 원리
- YAML 문법
- 패키지 설치와 서비스 관리의 기본 개념
- 인프라 자동화와 구성 관리의 차이

### 실습 과제
- 가상머신 2대 또는 컨테이너 기반 테스트 환경 준비
- SSH 키 생성 및 원격 접속 확인
- YAML 파일 3개 작성: 리스트, 딕셔너리, 중첩 구조

### 산출물
- 학습용 서버 목록
- SSH 접속 테스트 기록
- YAML 연습 파일

### 다음 단계 진입 기준
- SSH 키 기반으로 원격 서버에 접속할 수 있다.
- YAML 들여쓰기 오류를 스스로 찾을 수 있다.

## 1단계: 초급 입문
### 목표
- 앤서블의 기본 구성 요소와 실행 흐름을 이해한다.
- 단일 명령과 간단한 플레이북으로 서버 상태를 확인하고 변경할 수 있다.

### 핵심 주제
- 앤서블이란 무엇인가
- 제어 노드와 관리 대상 노드
- 인벤토리의 개념
- 애드혹 명령과 플레이북의 차이
- 모듈의 개념
- 멱등성의 의미
- `become`과 권한 상승

### 학습 순서
1. 앤서블 설치와 버전 확인
2. 인벤토리 파일 작성
3. `ping`, `command`, `shell`, `copy`, `file`, `service` 모듈 실습
4. 첫 번째 플레이북 작성
5. `ansible-playbook --check`와 `-vvv` 사용

### 실습 과제
- 인벤토리 파일로 로컬 테스트 대상 2대를 등록한다.
- 애드혹 명령으로 패키지 설치 가능 여부와 서비스 상태를 확인한다.
- Nginx 또는 Apache 설치 플레이북을 작성한다.
- 설정 파일 복사 후 서비스 재시작까지 포함한 기본 플레이북을 만든다.

### 권장 미니 프로젝트
- "웹 서버 1대 초기 세팅 자동화"
  - 패키지 설치
  - 서비스 시작 및 활성화
  - 간단한 HTML 파일 배포

### 산출물
- `inventory.ini`
- `site.yml`
- 실행 결과 로그 캡처 또는 기록

### 다음 단계 진입 기준
- 인벤토리와 플레이북의 차이를 설명할 수 있다.
- 애드혹 명령과 플레이북을 상황에 맞게 선택할 수 있다.
- 플레이북 실행 후 변경 사항을 검증할 수 있다.

## 2단계: 초중급 실전 기초
### 목표
- 반복 작업을 변수와 제어문으로 일반화할 수 있다.
- 여러 서버 그룹에 따라 다른 구성을 적용할 수 있다.

### 핵심 주제
- 변수 정의와 참조
- `group_vars`, `host_vars`
- 팩트 수집과 `ansible_facts`
- 조건문 `when`
- 반복문 `loop`
- `register`
- 핸들러
- 태그

### 학습 순서
1. 변수 파일 분리
2. 운영체제별 조건 분기
3. 여러 패키지/사용자/디렉터리 생성 자동화
4. 변경 발생 시 핸들러 실행 구조 이해
5. 특정 태스크만 실행하는 태그 전략 익히기

### 실습 과제
- 사용자 계정 여러 개를 반복문으로 생성한다.
- Ubuntu와 Rocky Linux에서 다른 패키지명을 조건문으로 처리한다.
- 템플릿과 핸들러를 이용해 웹 서버 설정 변경 시 재시작되게 만든다.
- `register` 결과를 출력해 조건 분기에 활용한다.

### 권장 미니 프로젝트
- "개발/운영 공통 웹 서버 베이스라인 구성"
  - 사용자 및 그룹 생성
  - 필수 패키지 설치
  - 방화벽 규칙 반영
  - 환경별 변수 적용

### 산출물
- 변수 분리된 플레이북 프로젝트
- 환경별 인벤토리
- 템플릿 파일 1개 이상

### 다음 단계 진입 기준
- 변수 우선순위가 왜 중요한지 설명할 수 있다.
- 조건문과 반복문을 섞어서 실무형 태스크를 작성할 수 있다.
- 핸들러가 필요한 이유를 이해한다.

## 3단계: 중급 구조화
### 목표
- 재사용 가능한 역할 기반 프로젝트를 설계할 수 있다.
- 코드 가독성과 유지보수성을 고려해 플레이북을 구조화할 수 있다.

### 핵심 주제
- Role 구조
- `defaults`, `vars`, `tasks`, `handlers`, `templates`, `files`, `meta`
- `include_tasks`, `import_tasks`
- 태스크 분리 전략
- 프로젝트 디렉터리 표준화
- Ansible Galaxy와 컬렉션 기초

### 학습 순서
1. 단일 플레이북을 역할 구조로 분해
2. 공통 역할과 서비스별 역할 구분
3. 기본값과 강제값의 차이 이해
4. 작업 파일 분리와 재조합
5. 컬렉션 도입과 외부 역할 검토

### 실습 과제
- 기존 웹 서버 플레이북을 `common`, `web`, `firewall` 역할로 나눈다.
- 역할마다 기본 변수를 분리한다.
- OS별 태스크 파일을 나누고 조건부 include를 적용한다.
- 컬렉션 1개를 도입해 관리형 자원 자동화를 실습한다.

### 권장 프로젝트
- "표준 리눅스 서버 베이스라인 Role 세트"
  - 공통 보안 설정
  - 시간 동기화
  - 사용자 정책
  - 로깅 기본 설정

### 산출물
- 역할 기반 디렉터리 구조
- 역할별 README 초안
- 공통 역할 재사용 예제

### 다음 단계 진입 기준
- 플레이북을 역할로 분해하는 기준을 설명할 수 있다.
- 변수 위치에 따라 유지보수성이 어떻게 달라지는지 이해한다.
- 여러 프로젝트에서 재사용 가능한 역할을 만들 수 있다.

## 4단계: 중고급 운영 자동화
### 목표
- 실제 운영 환경에서 필요한 보안, 예외 처리, 템플릿, 검증 전략을 적용할 수 있다.

### 핵심 주제
- Jinja2 템플릿
- `block`, `rescue`, `always`
- `failed_when`, `changed_when`
- `assert`
- `ansible-vault`
- 환경 분리 전략
- 체크 모드와 드라이런 해석
- 태그 기반 부분 배포

### 학습 순서
1. 설정 파일 템플릿화
2. 예외 처리 블록 작성
3. 실패 조건과 변경 조건 명시
4. 민감정보 암호화
5. 운영/스테이징/개발 환경 분리

### 실습 과제
- DB 접속 정보와 API 키를 `ansible-vault`로 관리한다.
- 설정 검증 명령을 포함한 템플릿 배포 플레이북을 작성한다.
- 서비스 재시작 전 설정 테스트를 수행하고 실패 시 롤백 흐름을 설계한다.
- `assert`로 필수 변수 누락을 사전에 차단한다.

### 권장 프로젝트
- "운영 환경용 안전한 웹 애플리케이션 배포 플레이북"
  - 템플릿 렌더링
  - 사전 검증
  - 서비스 재기동
  - 실패 처리
  - 비밀정보 분리

### 산출물
- Vault 적용 예제
- 템플릿 기반 설정 파일
- 예외 처리 포함 플레이북

### 다음 단계 진입 기준
- 운영 환경에서 평문 비밀정보를 피하는 이유를 설명할 수 있다.
- 실패 조건과 변경 조건을 제어할 수 있다.
- 드라이런 결과를 해석해 실제 변경 위험을 예측할 수 있다.

## 5단계: 고급 자동화 설계
### 목표
- 대규모 서버 집합과 복합 서비스 배포를 안정적으로 관리할 수 있다.
- 배포 전략, 성능, 가시성, 표준화를 고려한 자동화 설계를 할 수 있다.

### 핵심 주제
- 대규모 인벤토리 설계
- 동적 인벤토리
- 롤링 업데이트
- 시리얼 배포
- `delegate_to`, `run_once`
- 성능 최적화
- `strategy`, `forks`
- 테스트와 린트
- Molecule 기초

### 학습 순서
1. 정적 인벤토리 한계 이해
2. 클라우드 또는 VM 환경에서 동적 인벤토리 실습
3. 무중단 또는 저위험 배포 전략 설계
4. 역할 테스트와 린트 도입
5. 공통 품질 기준 문서화

### 실습 과제
- 서버 10대 이상을 가정한 그룹 구조를 설계한다.
- 동적 인벤토리 플러그인을 사용해 대상 목록을 자동 수집한다.
- `serial`을 사용한 롤링 배포 플레이북을 작성한다.
- `ansible-lint`와 Molecule로 역할 테스트를 구성한다.

### 권장 프로젝트
- "다중 웹 서버 + 로드밸런서 롤링 배포 자동화"
  - 배포 대상 순차 처리
  - 헬스체크
  - 실패 시 중단 조건
  - 배포 후 검증

### 산출물
- 대규모 인벤토리 설계 문서
- 롤링 배포 플레이북
- 테스트 자동화 설정

### 다음 단계 진입 기준
- 정적/동적 인벤토리의 장단점을 비교할 수 있다.
- 무중단 배포를 위한 앤서블 전략을 설명할 수 있다.
- 린트와 테스트를 자동화 파이프라인에 포함시킬 수 있다.

## 6단계: 전문가 단계
### 목표
- 조직 단위의 자동화 표준을 만들고, 배포 플랫폼 및 운영 프로세스와 통합할 수 있다.
- 앤서블을 단순 도구가 아니라 운영 체계의 일부로 설계할 수 있다.

### 핵심 주제
- Automation Controller 또는 AWX
- RBAC와 작업 템플릿
- 스케줄링과 승인 흐름
- CI/CD 연동
- GitOps 관점의 자동화 관리
- 감사 추적과 변경 이력
- 표준 Role/Collection 체계
- 보안 기준과 운영 거버넌스

### 학습 순서
1. 수동 실행 중심 운영의 한계 분석
2. 중앙 관리 플랫폼 도입
3. 권한 체계와 승인 정책 설계
4. Git 기반 변경 관리와 파이프라인 연계
5. 조직 표준 역할과 템플릿 정립

### 실습 과제
- AWX 또는 유사 환경에 프로젝트와 인벤토리를 등록한다.
- Job Template을 만들고 자격증명 및 권한을 분리한다.
- Git 저장소 변경 시 자동 배포 또는 점검 작업이 실행되게 구성한다.
- 공통 Role 품질 기준, 리뷰 기준, 버전 정책을 문서화한다.

### 권장 프로젝트
- "조직 표준 서버 프로비저닝 및 배포 자동화 플랫폼 설계"
  - 형상 표준화
  - 권한 통제
  - 실행 이력 추적
  - 운영 승인 절차
  - 품질 게이트 적용

### 산출물
- 운영 자동화 아키텍처 문서
- 중앙 실행 플랫폼 구성 문서
- 표준 역할/리뷰/배포 정책 문서

### 전문가 단계 진입 기준
- 여러 팀이 함께 쓰는 자동화 체계를 설계할 수 있다.
- 보안, 운영, 개발 요구사항을 동시에 만족하는 자동화 정책을 정의할 수 있다.
- 자동화 실패가 서비스 장애로 이어지지 않도록 통제 장치를 설계할 수 있다.

## 단계별 권장 학습 기간
- 준비 단계: 3일~1주
- 초급 입문: 1주~2주
- 초중급 실전 기초: 2주~3주
- 중급 구조화: 2주~4주
- 중고급 운영 자동화: 3주~5주
- 고급 자동화 설계: 4주~6주
- 전문가 단계: 6주 이상 지속 심화

## 공통 학습 습관
- 모든 플레이북은 테스트 환경에서 먼저 실행한다.
- `--check`와 실제 실행 결과 차이를 비교하며 기록한다.
- 실패 로그를 모아 원인을 분류한다.
- 하나의 큰 플레이북보다 역할 단위 재사용성을 우선한다.
- 비밀정보는 초반부터 Vault 또는 외부 비밀 저장소 사용 습관을 들인다.
- 실행 전후 검증 명령을 반드시 포함한다.

## 추천 실습 주제 순서
1. 패키지 설치 자동화

## 최종 포트폴리오 권장 구성
- 기본 서버 초기화 플레이북
- 역할 기반 웹 서비스 배포 프로젝트
- Vault 포함 운영 환경 프로젝트
- 롤링 배포 가능한 고급 배포 프로젝트
- 테스트와 린트가 포함된 역할 저장소
- AWX 또는 CI/CD 연동 자동화 설계 문서

## 최종 목표
이 로드맵의 끝에서는 다음을 할 수 있어야 한다.

- 서버 초기 설정을 표준화된 플레이북으로 자동화한다.
- 역할 기반으로 유지보수 가능한 앤서블 프로젝트를 설계한다.
- 보안, 검증, 롤백을 고려한 운영 자동화를 구현한다.
- 대규모 환경과 조직 단위 프로세스까지 고려한 자동화 체계를 설계한다.
